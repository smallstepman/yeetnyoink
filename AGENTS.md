# Introduction

**yeetnyoink** is a geometry-first window/pane orchestration system that provides seamless navigation across heterogeneous tiling environments. It unifies focus, move, and resize operations across:

- **Window Managers** (niri, i3, yabai, aerospace)
- **Terminal Emulators** (wezterm, kitty, foot, alacritty, ghostty, iTerm2)
- **Terminal Multiplexers** (tmux, zellij, wezterm mux, kitty native)
- **Editors** (neovim, emacs, vscode)
- **Browsers** (firefox, chrome)

Rather than hardcoding behavior, the system queries the current state to understand the topology (the spatial layout of windows/panes), applies policies read from user config + hardcoded per-adapter capabilities, and selects the appropriate action dynamically.

```
│                                                               │

│  (src/commands/)                                              │

│  - Delegates to orchestrator                                  │

                     │

│                     Engine Layer                              │

│  - Domain/adapter contracts                                   │

```

# Agent's notes

- Surprise encountered: `src/config.rs` currently appears inconsistent with the runtime call sites (`config::prepare`, `wm_adapter_override`, `app_adapter_override`, etc.) and includes schema/test content that does not obviously match the compiled API surface. Before doing architecture work, verify whether config code is mid-migration or partially generated to avoid debugging the wrong interface. After probing the user, I understood that the src/config.rs has recently been rebuilt and it's shape is more-or-less correct and stable, while the rest of the code needs to adapt to current state of config.rs
- Surprise encountered: after the orchestrator rewrite, app adapters' `move_decision/move_internal/move_out` were not being invoked by `move` at all in normal routing, so edge moves in terminal/editor panes silently fell back to WM moves (for niri this triggers `ConsumeOrExpelWindow*`). If `move` behavior seems WM-only, inspect `src/engine/orchestrator.rs` first.
- Surprise encountered: same-domain routing (`source.domain == target.domain`) originally skipped transfer/merge logic entirely and always delegated to WM move, which prevented expected "merge back" behavior for app windows of the same adapter (e.g. wezterm↔wezterm). If merge-back does nothing, inspect `execute_move` same-domain branch in `src/engine/orchestrator.rs`.
- Surprise encountered: wezterm `get-pane-direction` can be inconsistent across immediate back-to-back queries (decision says perpendicular neighbor exists, action query returns none). Rearrange needs a fallback target selection from same-tab pane list instead of assuming direction probes are stable.
- Surprise encountered: the historical WezTerm mux bridge/plugin path became stale: Rust wrote bridge commands under `$XDG_RUNTIME_DIR/yeetnyoink-wezterm-mux`, while `plugins/wezterm` polled `/tmp|$XDG_RUNTIME_DIR/yny-wezterm-mux`. Treat WezTerm as CLI-only unless you intentionally reintroduce and wire a single bridge protocol on both sides.
- Surprise encountered: app-level tear-out can produce the new window but still place it with default WM geometry unless orchestrator explicitly applies WM tear-out composition (`plan_tear_out`) after `move_out`. Directional placement (W/E/N/S) must be applied as a post-tearout WM step.
- Surprise encountered: `wezterm cli move-pane-to-new-tab --new-window` may leave WM focus on the source window immediately after tearout. If post-tearout placement logic assumes focus switched to new window, it will silently no-op; detect/focus the new window from WM snapshots first.
- Surprise encountered: new WM windows from tearout may not be visible in the very first post-action snapshot. A short retry/poll loop around `wm.windows()` is needed before concluding no new window exists.
- Surprise encountered: `execute_focus` can silently bypass app adapters if it only routes via WM topology. For terminal/editor internal pane focus to work, focus needs the same app-first handling pattern used by move.
- Surprise encountered: `snapshot_from_wm` currently assigns synthetic left-to-right rectangles from window list order (`x = idx * 1000`) rather than true WM geometry/adjacency. Any routing that relies on `find_neighbor` over that snapshot can pick non-neighbor windows (notably with multiple Emacs frames), so directional target selection should use WM-native directional probing until real geometry is available.
- Surprise encountered: emacs `merge_into` originally selected `(car (delq src ...))`, i.e. an arbitrary visible frame, and silently no-op'ed when no target existed. For reliable snap-back with multiple frames, pick the closest directional frame and raise an explicit error if no directional target exists.
- Surprise encountered: Emacs frame geometry (`frame-position`/pixel bounds) may not reliably reflect WM directional adjacency under some compositors; source-focused directional merge targeting can still choose the wrong frame. A robust approach is target-focused merge with a persistent per-frame ID captured in `prepare_merge`.
- Surprise encountered: Emacs `make-frame` + workspace switch can clone an existing multi-window layout into the torn-out frame; without an explicit `delete-other-windows` in the new frame, tear-out of one buffer can leave unexpected duplicated layouts across source/new frames.
- Surprise encountered: once merge preparation carries source identity (e.g., editor frame IDs), `MergePreparation` can no longer be `Copy`; treat it as owned state and consume it exactly once at merge execution.
- Surprise encountered: config loading is now driven by `etcetera` + optional CLI `--config`; tests should use `config::prepare_with_path(...)` (or save/restore `config::snapshot()`) to avoid picking up host-user config files from platform defaults.
- Surprise encountered: "default wezterm backend" tests are especially easy to contaminate with host-user config because `active_mux_provider(WEZTERM_HOST_ALIASES)` follows the loaded config and will happily resolve to tmux/zellij/kitty instead of native wezterm. Tests that assert default mux capabilities or attach-command behavior must load an explicit empty/minimal config before calling `config::prepare_with_path(...)`.
- Surprise encountered: Neovim process discovery can return the outer UI pid while the RPC socket belongs to a descendant `nvim --embed` child (for example parent pid `643699` with socket file `nvim.643700.0`). If `resolve_terminal_chain` logs multiple nvim descendants but chain depth stays at 1, make `Nvim::for_pid` probe descendant nvim pids for `NVIM[_LISTEN_ADDRESS]` / `nvim.<pid>.*` sockets instead of assuming the first pid owns the socket.
- Surprise encountered: when tmux reports the focused pane foreground as `nvim`, the old `resolve_terminal_chain` `"nvim"` branch still picked the *first shell with any nvim descendant* from the terminal process tree. With multiple nvim/tmux panes open, that often targeted the wrong editor instance even though tmux already knew the focused pane. Prefer `tmux.nvim_in_current_pane()` for the tmux-backed `"nvim"` path, and only fall back to shell descendant scans if that focused-pane lookup fails.
- Surprise encountered: Neovim `json_encode({...,'modified':&modified})` can decode `modified` as numeric `0/1` rather than JSON boolean `false/true`, so strict `bool` deserialization of snapshot JSON breaks tear-out before any real routing happens. Make snapshot parsing accept bool-like integers/strings.
- Surprise encountered: `expand('%:p')` is **not** enough to prove a Neovim buffer is file-backed; terminal buffers can come through as `term://...` with a non-empty path. Gate Neovim tear-out on `&buftype == ''` (or equivalent snapshot metadata) so terminal/special buffers fail explicitly instead of being relaunched as bogus file paths.
- Surprise encountered: `nvim --remote-expr "luaeval(...)"` expects the inner Lua chunk to be an *expression*, not a statement list. Raw `luaeval("local ...; return ...")` fails at runtime with `unexpected symbol near 'local'`; wrap smart-splits helper bodies as `(function() ... end)()`.
- Surprise encountered: the Neovim test harness keys fake `nvim` responses by sanitized command string. Long `--remote-expr` commands can collide when truncated, silently making one mocked command reuse another response; append a checksum/hash suffix when deriving response keys or the smart-splits tests will lie.
- Surprise encountered: terminal adapters returning `MoveDecision::Passthrough` were previously treated as “merge only, otherwise fall back to WM move”, which broke layered tear-out flows like `nvim buffer -> mux pane -> WM window`. If repeated edge moves should escalate outward when no same-adapter merge target exists, let the orchestrator attempt adapter `move_out` after passthrough-merge probing fails.
- Surprise encountered: introducing a dedicated `TerminalHost` trait duplicated most of `AppAdapter` metadata (`adapter_name/config_aliases/kind/capabilities`) with little extra value and increased cognitive load. Prefer keeping host identity/capabilities on `AppAdapter` and introducing new traits only for genuinely orthogonal concerns (e.g., mux topology provider behavior).
- Surprise encountered: during the apps→terminal_multiplexers move, leaving mux backend selection on `apps::wezterm::WeztermBackend::mux_provider` while mux internals moved out creates split ownership and accidental reverse coupling (`terminal_multiplexers` importing app aliases). Keep backend selection/helper entry points centralized in `adapters::terminal_multiplexers` and avoid importing app modules from mux modules.
- Surprise encountered: orchestrator move tests can fail in full parallel `cargo test` runs while passing individually due shared global config/env mutation across tests. If these assertions flap (`wm fallback` count mismatches), rerun with `-- --test-threads=1` and audit env/config-touching tests for isolation.
- Surprise encountered: `TerminalMultiplexerProvider::move_out` should return mux-level spawn args only; terminal launch prefix composition happens in the terminal host adapter (`apps::wezterm` delegates). In tmux provider paths, passing an empty launch prefix into internal move-out composition is intentional and avoids double-prefixing.
- Surprise encountered: `resolve_terminal_chain` shell disambiguation uses tpgid matching (`shell.tpgid comm == fg_base`) to pick the focused shell when multiple shells exist under a terminal. This fails when the fg process is running *inside* tmux (e.g. "node"), because the shell's tpgid points to the tmux client, not the inner process. When disambiguation yields `search_pid = None`, the resolver previously bailed to "terminal layer only" without ever trying tmux. Similarly, the `match fg_base` block only handled "tmux" and "nvim" — any other fg_base (e.g. "node", "python") fell through `_ => {}` with no tmux detection even when `search_pid` was found. Both cases now fall back to `find_descendants_by_comm(shell_pid, "tmux")` on each shell candidate.
- Surprise encountered: tmux adapter behavior became inconsistent after opening multiple wezterm windows attached to the same tmux session because many tmux queries/actions were targeted only by `session` (e.g. `display-message -t <session>`), which is ambiguous across clients and can return/use the wrong focused pane. Scope focused-pane reads and pane actions by the specific tmux client pid (via `list-clients` + `#{pane_id}`) to avoid false "source and target panes are the same" merge failures and incorrect internal focus/move routing.
- Surprise encountered: tmux discovery can silently pick the wrong process when code assumes “descendant tmux” is always a client. In tmux-launched windows, the terminal pid itself may be the tmux client, while descendants can include server/helper tmux processes; always consider root pid when `comm=="tmux"` and resolve candidates by trying `from_client_pid` across all candidates instead of trusting the first descendant.
- Surprise encountered: tmux format fields `#{pane_left}/#{pane_right}/#{pane_top}/#{pane_bottom}` are pane geometry coordinates, **not** neighbor pane IDs; using them as IDs causes bogus targets like `%67`. Also, older tmux versions may reject `swap-pane -L/-R` and require explicit `swap-pane -s <src> -t <dst>`, and `break-pane` must target source pane via `-s` (not `-t` pane id).
- Surprise encountered: tmux tearout that spawns `attach-session -t <same-session:window>` can make source/new clients appear mirrored because both attach to the same session state. For stable tearout/merge UX, create a grouped detached session (`new-session -Ad -s <new> -t <source_session>`) and attach the new terminal client to `<new>:<window_index>` so source and torn-out windows stay independent while sharing pane graph for `join-pane` merge.
- Surprise encountered: after tmux `swap-pane -s <src> -t <dst>`, focus may stay on position rather than following the moved source pane. If repeated directional move should continue from the same pane (e.g. `move west` twice: swap then tear-out), explicitly `select-pane -t <src>` after swap.
- Surprise encountered: after merge-back from a torn-out grouped tmux session, leaving the detached session alive keeps the torn-out terminal client attached and can show a duplicate mirrored window. Persist source session identity in merge preparation and `kill-session -t <detached_session>` after successful `join-pane` when merging into a different target session.
- Surprise encountered: probing kitty mux state with `pid == 0` (e.g. during chain-resolution tests without a real focused terminal pid) can trigger raw kitty remote-control escape traffic and slow/noisy test runs. Guard kitty foreground probes for invalid pid and skip CLI calls when pid is unset.
- Surprise encountered: `zellij action ...` can exit with status 0 even when it *did not* execute the action and printed “Please specify the session name…” (plus session list) to stderr/stdout. Treat that prompt as an error and resolve/pin the target `--session`, otherwise focus/move silently fall back to WM.
- Surprise encountered: zellij `move-focus <dir>` may return success even when focus did not move (edge/no-op). If neighbor probing always sends the opposite direction afterward, it can mutate focus away from the original pane (e.g., right no-op followed by left). Only issue the opposite-direction restore when the post-probe focused pane actually changed.
- Surprise encountered: `zellij action dump-layout` includes plugin panes, non-focused tabs, and templates; counting every `pane` line produces inflated pane counts and unstable move decisions. Parse only terminal leaf panes in the focused tab (fallback to legacy parsing only if focused-tab extraction fails).
- Surprise encountered: zellij 0.43.x includes `BreakPane/BreakPane{Left,Right}` in keybinding actions, but the `zellij action` CLI does not expose equivalent subcommands. Any edge tear-out implementation must therefore be a workaround (eg. floating + client attach) rather than a true CLI break-pane operation.
- Surprise encountered: a practical zellij tear-out workaround is "float focused pane + (optionally) hide floating panes in source tab + spawn a new terminal client attached to the same session"; this depends on zellij client/tab floating-visibility semantics and may still present shared-session coupling across clients.
- Surprise encountered: spawning `zellij attach <session> options --mirror-session false` did not reliably yield independent client view state in this flow; using a temporary zellij config (`mirror_session false`) for the spawned client (`zellij --config <file> attach <session>`) is a more reliable no-mirror hint.
- Surprise encountered: zellij break-pane functionality *is* exposed in the `zellij-tile` plugin API (`break_panes_to_new_tab` / `break_panes_to_tab_with_index`) even though it is absent from `zellij action` CLI; a pipe-triggered plugin is the viable non-keybinding route.
- Surprise encountered: some environments package Rust without wasm targets or rustup; building zellij plugins may require bootstrapping a rustup toolchain with `wasm32-wasip1` before producing `.wasm` artifacts.
- Surprise encountered: external zellij plugins require runtime permission grants (notably `ChangeApplicationState`) before break-pane API calls work; pipe-triggered plugins can otherwise no-op with success status, so request permissions in `load` and surface an explicit “grant plugin permission” error when post-break topology is unchanged.
- Surprise encountered: deriving plugin search root from `current_exe().parent().parent()` resolves to `<repo>/target` for local release binaries, so lookup misses `<repo>/plugins/...` unless cwd is repo root. Include `CARGO_MANIFEST_DIR` and walk executable ancestors when discovering `plugins/zellij-bridge` artifacts.
- Surprise encountered: `zellij action list-clients` can report `terminal_0` when invoked from non-client context; treating this as a real pane id makes break-pane no-op. Reject pane id 0, and let the plugin resolve focused pane from `ListClients`/`PaneUpdate` state with a retry path when no pane id is passed.
- Surprise encountered: zellij plugin permission cache lives in `~/.cache/zellij/permissions.kdl` (not under `~/.config`), keyed by plugin location string (usually the raw absolute path without `file:`). If a plugin starts requesting an additional permission later (eg. adding `ReadApplicationState` to an already-granted `ChangeApplicationState` plugin), `zellij pipe` can appear to hang waiting for a fresh permission prompt in the client.
- Surprise encountered: pointing `[runtime.zellij].break_plugin` at changing `/nix/store/<hash>/...` paths can invalidate previously granted permission cache entries and retrigger prompt/hang behavior. Prefer stable project/config plugin paths first and use the runtime config override only when needed.
- Surprise encountered: `zellij action pipe ... -- <payload>` may remain attached long enough to hit short command timeouts even when the plugin side effect already fired. Treat pipe timeout as indeterminate (kill child and continue) and decide success from post-action topology checks (`dump-layout` pane count), not from pipe process exit.
- Surprise encountered: CLI `zellij action ...` requests from non-client context are routed through the server’s “last active client” (for CLI clients) rather than a provided pane hint, so post-spawn tab switches can affect whichever client is currently active unless you deliberately sequence focus/client activation.
- Surprise encountered: zellij merge-back can be implemented through the plugin API using `break_panes_to_tab_with_index` (target active tab) even though there is no direct CLI `join-pane` equivalent; route merge through the pipe plugin instead of trying to synthesize it from `move-pane`/`stack-panes` CLI actions.
- Surprise encountered: trying to force torn-out-client tab selection via a spawned shell wrapper (`... zellij action go-to-tab-name ... & exec zellij attach ...`) is brittle because CLI actions may target last-active source client and query outputs can be stale/misaligned, producing invalid tab picks (eg `"layout {"`). Prefer direct attach and avoid post-spawn tab juggling from non-client context.
- Surprise encountered: for zellij tearout, the newly attached client follows whichever tab is active at attach time; if break uses `focus_new_tab=false`, source and torn-out clients often land on the same tab. A more reliable pattern is `break_panes_to_new_tab(..., focus_new_tab=true)` plus a delayed source-tab restore from plugin context (Timer/ListClients) after the new client appears.
- Surprise encountered: pipe-triggered zellij plugin state can be stale/empty at break time (eg no cached active tab or client count yet), causing immediate/early source-tab restore and re-coupling both clients onto one tab. Pass `source_tab_index` and `source_client_count` from adapter CLI probes into pipe args and restore only after `clients >= baseline + 1`.
- Surprise encountered: even after the new client is visible in `ListClients`, restoring the source tab immediately can still race client initialization and make both clients land on the source tab. Add a short post-attach grace timer before switching the source client back.
- Surprise encountered: in `zellij-tile` 0.43.1, `break_panes_to_tab_with_index` takes args as `(pane_ids, tab_index, should_float)`. Passing `(pane_ids, should_float, tab_index)` compiles nowhere and breaks nix builds; keep merge callsites aligned with shim signature.
- Surprise encountered: local repo plugin artifacts (`plugins/zellij-bridge/target/...wasm`) can silently shadow newer nix-built plugin outputs, and zellij logs will still look "successful" while running stale behavior. Plugin discovery should skip stale local artifacts (when `src/main.rs` is newer) and include `/nix/store/*yeetnyoink-zellij-break*/...wasm` candidates.
- Surprise encountered: the canonical hosted zellij plugin URL now comes from the rolling `zellij-plugin-latest` GitHub release. Keep `yny setup zellij`, runtime fallback URLs, and the release workflow aligned on the same tag and asset name.
- Surprise encountered: torn-out kitty→zellij attach clients may not expose `ZELLIJ_PANE_ID` anywhere in their process tree, so merge prep can fall back to ambiguous `zellij action list-clients` and target the wrong pane. Preserve the torn-out pane id explicitly in the spawned attach command environment and read that env var before any CLI fallback.
- Surprise encountered: already-torn-out kitty→zellij windows created *before* pane-id env propagation keep running plain `zellij attach ...`, so merge prep still sees no env marker even after rebuilding the binary. Add a plugin-based `query-pane-id` fallback (via `zellij action pipe` + `cli_pipe_output`, which requires `ReadCliPipes`) so stale windows can still resolve their current pane without relying on `list-clients`.
- Surprise encountered: successful passthrough merge at the app/mux layer does not automatically remove the now-empty WM window; without explicit orchestrator-side WM cleanup, zellij merge-back can leave an empty torn-out kitty window behind. After a successful passthrough merge, focus the target WM window and close the source WM window by id.
- Surprise encountered: terminals without their own pane-control plane (eg. Foot) do **not** need a dedicated `terminal_multiplexers::<host>` backend. Model them as terminal app adapters that delegate directly to the configured mux provider (tmux/zellij/etc.), and only add host-specific wiring for app IDs, launch prefix, and default mux policy.
- Surprise encountered: plain `zellij` clients launched inside generic terminals (eg. Foot) may expose **no** `ZELLIJ_SESSION_NAME` on the client process and no session-bearing client cmdline (`cmdline == "zellij"`), while the detached server owns the named `/run/user/.../zellij/.../<session>` socket. In that case, descendant/env/cmdline discovery is insufficient; resolve the session by matching the client’s unix socket inodes to the server socket path via `ss -xnp` peer output, then cache the discovered session for the terminal pid.
- Surprise encountered: Wayland terminal app IDs are not consistently lowercase/program-name-shaped. Alacritty’s default app_id is `Alacritty` (capital A), while Ghostty’s default is `com.mitchellh.ghostty` (and debug builds may use `com.mitchellh.ghostty-debug`). When adding “generic terminal host” adapters, verify actual compositor-visible app IDs instead of assuming the executable name.
- Surprise encountered: terminal-host capability tests that call `AppAdapter::capabilities()` are config-sensitive because host adapters delegate to the currently selected mux backend. If a test doesn’t load an explicit per-test config, a kitty/wezterm host assertion can silently reflect ambient host-user mux policy (eg. kitty resolving to tmux and losing resize/rearrange), so capability tests must pin config first.
- Surprise encountered: Ghostty tear-out windows can appear noticeably later than Foot/Alacritty and may leave WM focus on the source window long enough for composed placement to no-op; poll for the new WM window and explicitly refocus it before applying `plan_tear_out`, or niri will leave the torn-out window in its default right-side insertion slot.
- Surprise encountered: kitty remote-control pane movement is split across two surfaces: `kitty @ action ...` for pane actions (`move_window left/right/up/down`, `layout_action move_to_screen_edge left/right/top/bottom`) and dedicated RC subcommands for structural ops (`detach-window`, `resize-window`). `kitty @ move-window` is not a real command, and the direction vocabulary differs between neighbor matchers (`left/right/top/bottom`) and `move_window` actions (`left/right/up/down`).
- Surprise encountered: kitty native mux can work from inside a kitty shell without a socket (via controlling tty) but **cannot** be driven from detached WM keybindings unless kitty exposes a remote-control address. If `KITTY_LISTEN_ON` is absent and stderr contains `/dev/tty`, instruct the user to add `allow_remote_control socket-only` plus `listen_on unix:@kitty-{kitty_pid}` to `kitty.conf` and restart kitty.
- Surprise encountered: kitty `ls --output-format json` on 0.44.x exposes more layout metadata than expected (`layout`, `enabled_layouts`, `groups`, `layout_state`), but it still is not a general-purpose WM-grade geometry source. For move decisions, the generic “perpendicular neighbors => Rearrange” heuristic is still too eager; prefer TearOut when there is no direct neighbor and reserve rearrange for explicit/manual use.
- Surprise encountered: kitty merge into a single-pane target tab inherits that tab’s current layout, so `detach-window --target-tab ...` plus post-merge `move_window`/`move_to_screen_edge` can leave a west/east snap-back stacked as top/bottom. A more reliable pattern is to set the target tab to `goto-layout splits:split_axis=horizontal|vertical` **before** `detach-window`, then do best-effort edge placement after the merge.
- Surprise encountered: tests that exercise WezTerm bridge/default mux behavior can silently inherit whatever explicit config / cached config a previous test loaded; make harnesses pin an explicit wezterm config (and restore `config::snapshot()` on drop) or “bridge by default” tests may unexpectedly take the direct CLI merge path and fail on missing mocked `list --format json` responses.
- Surprise encountered: VS Code's stable extension API still does not expose the full editor-group split tree, but the internal command surface *does* expose `vscode.getEditorLayout`/`vscode.setEditorLayout`, and those commands are callable through `vscode-remote-control`. The layout response still omits the active group, so adapters can infer group count/topology shape but not do fully edge-accurate directional focus/move decisions without an extra state-export command.
- Surprise encountered: `cargo test` only refreshes debug artifacts; if the user is invoking `target/release/yny`, they may still be exercising an old release binary that predates your source changes. Rebuild with `cargo build --release` before diagnosing runtime logs from `target/release/yny`.
- Surprise encountered: the current VS Code adapter now persists a per-window focus heuristic on disk (`$XDG_RUNTIME_DIR/yeetnyoink/vscode-state.json`, override with `[runtime.vscode].state_file`) because the bridge cannot report the active editor group directly. If focus/move behavior looks stale after manual VS Code navigation or layout edits, inspect/reset that cache and verify the layout signature logic before debugging the orchestrator.
- Surprise encountered: VS Code remote-control focus commands can resolve before the workbench visibly settles, so rapid repeated `focus` calls can outrun the persisted heuristic and escape to the WM too early. Keep a short settle window (`[runtime.vscode].focus_settle_ms`) plus conservative pending-state routing so repeated directional focus stays inside VS Code until the optimistic editor position is confirmed.
- Surprise encountered: `vscode-remote-control` action commands (eg `workbench.action.focusLeftGroupWithoutWrap`) may never send a websocket reply at all; if `run_command` reuses the same request/response path as `vscode.getEditorLayout`, it burns the full 1s socket read timeout on every focus/move. Treat layout/state queries as request/response, but send action commands fire-and-forget after a successful websocket send.
- Surprise encountered: distinct VS Code windows can still resolve to the same process pid, so pid-based “wait until source window disappears” logic is invalid for merge-back and can burn the full restore polling loop even when the command succeeded. Let the orchestrator close the specific source WM window by id after a short settle instead of polling `/proc/<pid>` or `getEditorLayout(pid)` for disappearance.
- Surprise encountered: VS Code `workbench.action.moveEditorTo{Left,Right,Above,Below}Group` moves only the active editor tab, not the whole editor group/pane. Using those commands for pane movement can duplicate or swallow neighboring groups (`A | B` turning into `A | A | B` or collapsing into one group). For pane-level movement, use `workbench.action.moveEditorGroup{Left,Right,Up,Down}` instead.
- Surprise encountered: the *intended* VS Code topology semantics in this project are not “move panes/groups around”; they are “focus editor groups, but move the active tab between groups.” For this adapter, internal directional move should therefore use `moveEditorTo{Left,Right,Above,Below}Group`, while tear-off scope decides whether out-of-app movement uses `moveEditorToNewWindow` (buffer/tab) or `moveEditorGroupToNewWindow` (editor group). If you see conflicting advice elsewhere, prefer the project-specific semantics over the generic “pane movement” interpretation.
- Surprise encountered: `vscode-remote-control` special-cases `vscode.open`/`vscode.openFolder` by converting a string first argument into `vscode.Uri.file(...)`, which makes target-focused VS Code merge viable without a custom extension. However, it still does not return active editor identity, so buffer-level merge preparation needs an out-of-band capture path (currently `copyFilePath` + clipboard read/restore) rather than expecting a websocket response payload.
- Surprise encountered: VS Code still exposes no direct command to list the tabs in the active editor group, so window-scope merge preparation has to *approximate* group capture by iterating `workbench.action.openEditorAtIndex` and sampling `copyFilePath` from the clipboard. This is good enough for normal file-backed tabs, but identical consecutive tabs are still ambiguous; if group merge drops/repeats a duplicate tab, inspect the `capture_active_group_paths` heuristic first.
- Surprise encountered: `copyFilePath` can silently no-op for terminal/non-file tabs and leave the clipboard unchanged, so naive “run command, then read clipboard” capture can return a stale previous file path. Write a unique probe token to the clipboard first, and treat an unchanged probe as “no file-backed editor exposed” before deciding how VS Code merge prep should behave.
- Surprise encountered: VS Code terminal support is much flatter than editor-group support: the built-in command surface has useful actions (`workbench.action.terminal.focusPrevious/focusNext`, `...moveIntoNewWindow`, `...moveToTerminalPanel`) but still no bridge-visible terminal layout/count query. Keep terminal instance management opt-in via config and treat it as best-effort linear terminal handling rather than true pane-topology reconstruction.
- Surprise encountered: `tracing-chrome`/`tracing-flame` layer types are inferred against the *current* subscriber stack shape, so prebuilding or boxing them too early (especially with mixed fmt writer types) leads to type-mismatch explosions. Build profiling layers in the same branch/order as the final subscriber composition instead of trying to stash `Registry`-typed layers for later.
- Surprise encountered: `src/adapters/terminal_multiplexers/zellij.rs` previously treated `send_text_to_pane` as unsupported, but current zellij CLI actually exposes `zellij action write-chars` plus `zellij action write 13` for focused-pane input. For Neovim smart-splits tear-out, zellij can launch the new `nvim --listen ...` command as long as the split leaves the target pane focused (or the adapter explicitly verifies/follows focus before writing).
- Surprise encountered: app integrations now default to **disabled** unless their own matching profile sets `enabled = true`; the old “unmatched adapters stay enabled” assumption is no longer valid. For terminal-hosted editors (notably Neovim), nested discovery can still be re-enabled per editor through `app.editor.<editor>.ui.terminal.app = "<host>"`, so if terminal chain resolution seems unexpectedly off, inspect `section_enabled_from`, `terminal_chain_enabled_from`, and the editor UI helpers in `src/config.rs` before debugging the orchestrator.
- Surprise encountered: current WezTerm `cli list --format json` output may omit `foreground_process_name` entirely and only expose `tty_name`, which makes `active_foreground_process()` return `None` unless you derive the foreground job from `/proc` using that tty. When explicit WezTerm mux routing mysteriously falls back to WM/tmux with `fg_hint=None`, inspect `src/adapters/terminal_multiplexers/wezterm.rs` tty fallback first, and keep tmux fallback gated to configs that actually selected tmux.
- Surprise encountered: `MergeExecutionMode::TargetFocused` passthrough merges can still silently behave like terminal-level moves if the orchestrator probes the correct target window but then calls `merge_into_target()` on the **source** adapter instance. Stateful adapters like Neovim carry source-specific RPC/socket state, so target-focused merges must resolve the matching adapter from the target window's chain and execute the merge there.
- Surprise encountered: Neovim target-side merge automation is much less brittle when the multi-step edit/split logic lives inside `luaeval("(function() ... end)()")` rather than raw Vimscript expression composition. The raw `execute(...), cursor(...)` `--remote-expr` approach hit runtime `Vim:E110: Missing ')'` in real merge-back flows even though simpler unit mocks passed.
- Surprise encountered: layered Neovim merge-back from a torn-out mux pane does **not** cross a WM boundary, so WM directional probing alone will miss the real target and fall through to terminal pane swaps. For target-focused editor merges, probe the immediate outer adapter(s) in-place first (focus adjacent pane within the same terminal, re-resolve the chain, then merge there) before falling back to WM-level directional target search.
- Surprise encountered: torn-out Neovim merge-back can trip swapfile warnings if the target instance tries to `:edit` the file before the source torn-out instance has fully quit and removed its swap file. Capture `swapname()` during merge prep, quit the torn-out source first, and wait briefly for both RPC shutdown and swapfile disappearance before opening the buffer in the target instance.
- Surprise encountered: bare `swapname()` is not a valid no-arg probe in current Neovim remote expressions; it fails with `Vim:E119: Not enough arguments for function: swapname`. Use a buffer-aware Lua expression such as `vim.fn.swapname(vim.fn.bufnr('%'))` when capturing swap paths for merge preparation.
- Surprise encountered: Neovim tear-out must delete the source buffer, not merely close the source split. `:close` alone leaves the moved buffer loaded in the original instance, so tear-out cleanup should close the window and then `bdelete!` the captured current buffer from the source editor.
- Surprise encountered: after tearing Neovim out into a sibling mux pane, multiple shell children under the terminal can each have an `nvim` descendant, so old chain-resolution heuristics (`tpgid`/first descendant match) may resolve the *original* instance instead of the focused torn-out one. Preserve mux pane `tty_name` in `TerminalPaneSnapshot` and prefer the shell whose controlling tty matches the focused host pane tty before descendant-based fallback, or merge prep will act on the wrong Neovim and trigger swap warnings / pane-swap fallback.
- Surprise encountered: Firefox/LibreWolf browser tab control now uses a WebExtension + native-messaging host (`plugins/firefox-bridge`, `yeetnyoink-firefox-host`). Firefox native messaging still needs a background script/page (not a service worker), Firefox and LibreWolf need separate host-manifest paths, and release builds still need a Mozilla-signed XPI before policy-based force-install will work cleanly.
- Surprise encountered: Firefox/LibreWolf native-messaging manifests cannot point directly at the main `yny` CLI binary, because the browser can only launch an executable path while the CLI only enters host mode as `yny browser-host firefox`. For manual/non-Nix installs, generate a wrapper or dedicated host binary in the manifest; otherwise the browser-launched process exits immediately and can leave a stale `firefox-bridge.sock`.
- Surprise encountered: Firefox and Chromium browser bridges diverge at the extension layer even though the Rust socket/native-host protocol is shared. Keep the Firefox bridge on MV2/background-page semantics, but use a separate MV3 service-worker extension + `allowed_origins` manifests for Chromium/Brave, and make sure the Firefox manifest `gecko.id` stays aligned with Home Manager's `allowed_extensions` entry or native messaging silently breaks.
- Surprise encountered: separate browser-native host behavior does **not** need extra Rust binaries under `src/bin/`; this codebase/user expects browser host entrypoints to stay out of tree. Prefer tiny Home Manager/Nix wrapper scripts that exec `yny browser-host <firefox|chromium>`, while keeping browser-specific Rust logic inside `src/adapters/apps/`.
- Surprise encountered: on macOS, `accept()` on a nonblocking `UnixListener` may produce a stream that inherits nonblocking mode, unlike Linux where the accepted stream defaults to blocking. Test harnesses using fake Unix socket bridges (e.g., browser tests) must explicitly call `stream.set_nonblocking(false)` on accepted connections to avoid `WouldBlock` errors on `read_line`.
- Surprise encountered: `src/engine/runtime.rs` was 100% Linux-specific (used `/proc` filesystem extensively). For macOS support, this was split into `src/engine/runtime/` with platform-specific modules: `linux.rs` (existing /proc code) and `macos.rs` (shell-out to `ps`, `pgrep`, `lsof`). Cross-platform code should use the re-exports from `src/engine/runtime/mod.rs`.
- Surprise encountered: macOS has a 255-byte filename limit that is shorter than Linux. Test harnesses that use command strings as filenames (e.g., zellij mock responses) must truncate keys to avoid `InvalidFilename` errors. The shell script and Rust code must use the same truncation logic for response lookup to work.
- Surprise encountered: macOS `ps eww` output truncates long environment variables, so tests that compare full PATH values will fail. Use prefix matching instead of exact equality for environment variable assertions.
- Surprise encountered: the `focus-or-cycle` command is niri-specific (uses `niri_ipc::Window` type directly). It is conditionally compiled for Linux only via `#[cfg(target_os = "linux")]` in `src/commands/mod.rs` and `src/main.rs`.
- Surprise encountered: `niri-ipc` dependency is Linux-only and must be gated with `[target.'cfg(target_os = "linux")'.dependencies]` in `Cargo.toml`. The niri and i3 window manager modules are also Linux-only.
- Surprise encountered: WmBackend enum defaults to Yabai on macOS (changed from Aerospace). Aerospace adapter is declared in config but not implemented; if you see "unknown window-manager adapter override 'aerospace'" errors, the config is requesting the unimplemented Aerospace backend.
- Surprise encountered: macOS process tree walking must use a cached approach - spawning `ps`/`pgrep` for every process lookup resulted in 500ms+ per chain resolution (vs instant `/proc` reads on Linux). The solution uses a thread-local `ProcessTableCache` that fetches all process info in one `ps -A -o pid=,ppid=,comm=` call and remains valid for 100ms. This brought chain resolution from ~500ms to ~0.02ms.
- Surprise encountered: `cargo fmt --all --check` currently reports unrelated formatting drift in files outside the active feature area (recent examples included `src/engine/browser_native.rs`, `src/engine/kitty_setup.rs`, `src/engine/mod.rs`, `src/engine/zellij_setup.rs`, and `src/main.rs`). If you only touched a scoped subset of files, prefer formatting those files directly with `rustfmt` to avoid accidental repository-wide churn.
- Surprise encountered: WezTerm host-tab `split-pane --move-pane-id` did not reliably leave focus on the moved pane, and using the raw movement direction for cross-tab placement put the pane on the far edge of the target tab. For host-tab transfers, place using `dir.opposite()` and explicitly `activate-pane --pane-id <moved>` after the split so focus follows the moved pane.
- Surprise encountered: the `focus` command does **not** need runtime transfer domains, but `commands.run_action` used to load them eagerly before action execution. On niri empty workspaces this called `focused_window()` too early, failed with `no focused window`, and prevented native `FocusWindowOrWorkspaceUp/Down` fallback. Keep focus domain loading skipped, and treat `no focused window` in app-focus preamble as “no app context; fall back to WM.”
